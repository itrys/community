# ITrys 治理文档（Governance）

> **透明是信任的基石，规则是协作的保障。**
> 本文档定义 ITrys 组织的治理结构、决策流程与项目管理制度。

---

## 1. 治理原则

| 原则 | 说明 |
|------|------|
| **开放透明** | 所有治理决策公开记录，社区可追溯 |
| **贡献驱动** | 影响力来自贡献，而非头衔或职位 |
| **中立公正** | ITrys 不偏袒任何商业实体，保持平台中立性 |
| **社区共识** | 重大决策须经社区讨论，尽可能达成共识 |
| **可持续运营** | 治理设计须兼顾长期可行性与财务可持续性 |

## 2. 组织角色

### 2.1 角色层级

```
┌─────────────────────────────────────┐
│          Advisory Board             │   顾问委员会（外部专家）
├─────────────────────────────────────┤
│          Steering Committee         │   指导委员会（核心决策）
├─────────────────────────────────────┤
│     Project Management Committee    │   项目管理委员会（项目治理）
├─────────────────────────────────────┤
│     Maintainers / Contributors      │   维护者 / 贡献者
└─────────────────────────────────────┘
```

### 2.2 角色定义

#### 顾问委员会（Advisory Board）

- **构成**：3–5 名外部专家（法律、开源治理、商业策略等领域）
- **职责**：
  - 为重大战略决策提供专业建议
  - 审核治理规则的修订
  - 仲裁无法在社区内解决的争议
- **任期**：2 年，可连任
- **产生方式**：由指导委员会提名，社区投票确认

#### 指导委员会（Steering Committee）

- **构成**：5–7 名成员
- **职责**：
  - 制定组织战略方向与年度计划
  - 审批新项目入驻申请
  - 管理组织资金与资源分配
  - 任命项目管理委员会成员
  - 处理严重行为准则违规
- **任期**：1 年，可连任
- **产生方式**：
  - 创始期（首年）由发起人指定
  - 此后每年由活跃贡献者投票选举
- **退出机制**：连续 3 个月未参与委员会活动视为自动退出

#### 项目管理委员会（Project Management Committee, PMC）

- **构成**：每个入驻项目 1 名 PMC 代表 + 指导委员会指派 1 名联络人
- **职责**：
  - 审核项目入驻的技术合规性
  - 监督项目健康度（活跃度、合规性）
  - 协调跨项目协作
  - 管理项目生命周期（孵化 → 活跃 → 归档）
- **会议频率**：每月 1 次，公开记录

#### 维护者（Maintainer）

- **定义**：对项目有合并权限的核心贡献者
- **职责**：
  - 代码审查与合并
  - Issue 分类与响应
  - 版本发布管理
  - 确保项目符合 ITrys 准入标准
- **产生方式**：由项目现有维护者提名，PMC 确认
- **退出机制**：连续 6 个月无贡献视为休眠，合并权限暂停（可申请恢复）

#### 贡献者（Contributor）

- **定义**：提交过被合并的 PR/文档/设计等任何贡献的人
- **权利**：
  - 参与社区讨论与投票
  - 选举指导委员会成员
  - 申请成为维护者
- **义务**：遵守行为准则与 DCO 签署要求

## 3. 决策流程

### 3.1 决策类型与流程

| 决策类型 | 示例 | 决策方式 | 最低参与人数 | 通过条件 |
|----------|------|----------|-------------|----------|
| **日常决策** | Bug 修复方向、文档更新 | 维护者自行决定 | 1 名维护者 | 无异议即可执行 |
| **项目决策** | 新功能引入、版本规划 | 项目维护者共识 | ≥2 名维护者 | 简单多数 |
| **组织决策** | 新项目入驻、预算分配 | 指导委员会投票 | ≥3 名委员 | 简单多数（≥50%） |
| **重大决策** | 治理规则修订、组织方向调整 | 社区 RFC + 投票 | ≥5 名委员 + 社区公示 | 2/3 多数 |

### 3.2 RFC（Request for Comments）流程

所有重大变更须通过 RFC 流程：

```
1. 提案起草 → 2. 社区讨论（≥7 天） → 3. 修订迭代 → 4. 投票表决 → 5. 公布结果
```

**详细步骤**：

1. **提案起草**：任何贡献者可在 `community` 仓库提交 RFC Issue（使用 `rfc` 标签）
2. **社区讨论**：公示期不少于 7 天，所有人均可发表意见
3. **修订迭代**：提案者根据反馈修订方案，重大修订重新计时
4. **投票表决**：
   - 指导委员会成员投正式票
   - 贡献者可投建议票（仅供参考，不计入正式结果）
5. **公布结果**：在 Issue 中公布投票结果与执行计划

### 3.3 懒人共识（Lazy Consensus）

对于日常决策，若无人在 **72 小时**内提出异议，视为共识达成。

- 任何人可提出异议，异议须附理由
- 有异议时，升级为讨论或投票流程

## 4. 项目准入标准

### 4.1 基本要求

入驻 ITrys 的项目必须同时满足以下条件：

| # | 要求 | 说明 |
|---|------|------|
| 1 | **开源许可证** | 使用 [OSI 批准](https://opensource.org/licenses) 的开源许可证，推荐 MIT 或 Apache-2.0 |
| 2 | **可独立构建运行** | 提供完整的构建文档，CI 可通过，核心功能可用 |
| 3 | **非演示版** | 开源版必须功能完整，不得故意禁用关键 API 或设硬限制 |
| 4 | **许可证合规** | 所有依赖项许可证兼容，无法律风险 |
| 5 | **代码归属明确** | 提交者须签署 DCO（Developer Certificate of Origin） |
| 6 | **活跃维护承诺** | 至少 1 名维护者承诺持续维护（6 个月内） |

### 4.2 派生项目（Forked Projects）额外要求

- 已完成"去 fork 化"（在托管平台显示为独立仓库）
- 包含显著的原创性增强（新功能、国产化适配、性能优化等）
- 在 README 中致谢原项目
- 不接受仅做微小修改或未活跃维护的派生项目

### 4.3 商业模式声明

若项目采用商业友好模式（Open Core、SaaS、双许可等），须在入驻时：

- 明确声明商业模式与开源/付费边界
- 提供商业功能对比表
- 确保开源版符合"可构建、可运行、功能完整"原则

## 5. 项目生命周期

```
  ┌──────────┐     审核通过      ┌──────────┐     6个月无活动     ┌──────────┐
  │  申请中   │ ──────────────→  │   孵化期   │ ──────────────→  │   归档    │
  │ Applying │                  │ Incubating│                   │ Archived │
  └──────────┘                  └─────┬─────┘                   └──────────┘
                                      │ 达到活跃标准
                                      ▼
                                ┌──────────┐
                                │   活跃期   │
                                │  Active   │
                                └──────────┘
```

### 5.1 各阶段说明

| 阶段 | 持续时间 | 要求 | 权益 |
|------|----------|------|------|
| **申请中** | ≤ 30 天 | 提交入驻申请，填写项目信息 | 获得指导与反馈 |
| **孵化期** | 3–6 个月 | ≥1 名活跃维护者；每月至少 1 次提交；完成 SBOM 生成 | 获得 `ITrys Incubating` 徽章；社区推广支持 |
| **活跃期** | 无限期 | 维持活跃维护；年度合规自查 | 获得 `ITrys Certified` 徽章；优先推荐；认证报告发布 |
| **归档** | — | 连续 6 个月无维护活动 | 项目标记为归档状态；保留所有历史记录；可申请重新激活 |

### 5.2 退出机制

- **主动退出**：在 `community` 仓库提交退出 Issue，即日生效
- **被动归档**：连续 6 个月无维护活动，PMC 通知后 14 天无响应则自动归档
- **违规清退**：严重违反行为准则或开源合规要求，经指导委员会 2/3 投票可强制清退

> 📌 退出后，项目代码版权仍归原作者所有，ITrys 不保留任何知识产权。

## 6. 行为准则执行

### 6.1 执行主体

- **日常事务**：项目管理委员会处理
- **严重违规**：指导委员会处理
- **最终仲裁**：顾问委员会

### 6.2 处理流程

```
举报 → 初步评估（3 个工作日内）→ 调查（7 个工作日内）→ 裁决 → 执行
```

### 6.3 处罚措施

| 级别 | 违规行为 | 处罚 |
|------|----------|------|
| **提醒** | 无意冒犯、轻微不当言论 | 私下警告 |
| **警告** | 重复轻微违规 | 公开警告 + 行为改进要求 |
| **限制** | 较严重违规 | 限时禁言/禁止提交 PR（7–30 天） |
| **驱逐** | 严重歧视、骚扰、威胁 | 永久禁止参与所有 ITrys 项目 |

## 7. 知识产权与合规

### 7.1 DCO（Developer Certificate of Origin）

- 所有提交须包含 `Signed-off-by:` 行，表示贡献者确认：
  - 贡献为本人原创，或有权提交
  - 贡献以项目声明的许可证提供
- Git 提交时使用 `git commit -s` 自动添加

### 7.2 版权归属

- **项目代码版权归原作者或其所属组织所有**
- ITrys 不主张任何知识产权，不索取 CLA
- 社区贡献者保留其贡献的版权

### 7.3 合规检查

ITrys 为入驻项目提供自动化合规检查：

| 检查项 | 工具/方式 | 频率 |
|--------|----------|------|
| SBOM 生成 | SPDX 格式自动生成 | 每次版本发布 |
| 许可证兼容性 | 自动化扫描依赖树 | 每月 |
| 代码签名检查 | DCO 验证机器人 | 每次 PR |

## 8. 财务与可持续性

### 8.1 资金来源

| 来源 | 说明 | 占比上限 |
|------|------|----------|
| **企业赞助** | 无附加条件捐赠，不影响项目治理 | 单一企业不超过年度预算 30% |
| **社区捐赠** | 个人或组织自愿捐赠 | 无上限 |
| **服务收入** | 合规认证高级服务、培训、咨询 | 无上限 |
| **基金会资助** | 申请开源基金会或政府项目资助 | 无上限 |

### 8.2 资金使用

- 基础设施运营（CI/CD、托管等）
- 社区活动与推广
- 合规工具开发与维护
- 指导委员会批准的其他支出

### 8.3 财务透明

- 每季度在 `community` 仓库发布财务报告
- 所有单笔超过 ¥1,000 的支出须公开说明用途
- 年度预算由指导委员会审批，社区可提意见

## 9. 治理规则修订

### 9.1 修订流程

1. 任何贡献者可提议修订，通过 RFC Issue 提交
2. 社区讨论期不少于 **14 天**
3. 指导委员会投票，**2/3 多数**通过
4. 顾问委员会审核（确保与组织宗旨一致）
5. 公布生效

### 9.2 紧急修订

遇到法律合规等紧急情况时：

1. 指导委员会可启动紧急修订流程
2. 讨论期缩短至 **3 天**
3. 须 **全体委员同意**
4. 生效后须在 30 天内进行社区回顾
